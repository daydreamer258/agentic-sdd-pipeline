# 项目介绍说明

## 项目是什么

`agentic-sdd-pipeline` 是一个围绕 **SDD（Spec-Driven Development，规范驱动开发）** 构建的轻量级 agentic 编程工作流原型项目。

它的核心目标不是做一个“会自动写代码的聊天机器人”，而是做一套**可运行、可演进、可验证**的工程工作流，让 AI 能按照更接近真实软件工程的方式工作。

这个项目当前重点探索的是：

- 如何把 **SDD** 方法论落到实际仓库中
- 如何在只有 **Skills / Subagents / Hooks**，但没有复杂 Agent Teams 的条件下，搭建一个可用的 agentic workflow
- 如何让 AI 不只是“回答问题”或“生成片段”，而是真正按照阶段推进：
  - intake
  - spec
  - plan
  - tasks
  - implement
  - validate

---

## 项目解决什么问题

传统的 prompt-to-code 工作方式有一个明显问题：

- 需求一模糊，AI 就会开始猜
- 约束一分散，架构就容易漂
- 任务一变大，结果就越来越不稳定

这个项目试图解决的就是：

> **如何把 AI 编程从“凭感觉写”，推进成“按工件、按阶段、按约束稳定地写”。**

所以这里强调的不是“更强的模型幻觉”，而是：

- 更明确的阶段划分
- 更明确的工件结构
- 更明确的运行状态
- 更明确的 prompt layer
- 更明确的 backend execution handoff

---

## 当前采用的核心思想

这个项目当前的核心思想可以概括成一句话：

> **用 artifacts、runtime、prompt layer 和 backend consumer，把 SDD 早期阶段真正跑起来。**

也就是说，这个系统并不是把所有逻辑都塞进一条 prompt，而是拆成几层：

### 1. Artifact 层
围绕这些标准工件推进工作流：

- `00-intake.md`
- `01-spec.md`
- `02-plan.md`
- `06-tasks.md`
- `07-implementation-log.md`
- `08-validation.md`
- `09-retrospective.md`
- `state.json`

### 2. Runtime 层
负责：

- 初始化 feature
- 推进 stage
- 生成状态
- 调度 handler
- 记录工件完成情况

### 3. Prompt Layer
负责：

- 为每个 stage 提供 Skill prompt
- 为每种角色提供 Subagent prompt

### 4. Execution Integration 层
负责：

- 生成 stage bundle
- 把 runtime state 和 prompt layer 连接起来
- 把 stage 交给真实 backend 执行

### 5. Backend Consumer 层
当前已经接入：

- Claude Code CLI

并且已经验证可以真实消费以下阶段：

- `spec`
- `plan`
- `tasks`
- `validate`

## 当前已经支持哪些能力

截至目前，这个项目已经支持：

### 方法论与设计
- SDD / agentic workflow 研究与整理
- 系统设计文档
- artifact model
- stage rules
- runtime wiring 设计
- prompt layer 设计

### 运行时能力
- 初始化 feature 目录
- 生成标准工件模板
- 维护 `state.json`
- 生成 `.runtime/<stage>.bundle.md`
- 生成 backend handoff prompt 文件
- 运行 stage handler
- 完成 artifact 后更新状态
- validation 后更新状态

### Prompt Layer
- `sdd-intake`
- `sdd-spec`
- `sdd-plan`
- `sdd-tasks`
- `sdd-validate`

以及这些角色 prompt：
- `spec-writer`
- `planner`
- `task-decomposer`
- `validator`

### Backend Consumer
当前已经跑通：
- `spec -> 01-spec.md`
- `plan -> 02-plan.md`
- `tasks -> 06-tasks.md`
- `validate -> 08-validation.md`

### Auto Workflow
当前已经支持：
- 单独执行某个阶段
- 执行一个阶段区间
- 执行一个早期完整流程（默认 `intake -> tasks`）

### 真实 feature 验证
已经完成：
- `003-feature-summary-cli`
- `005-claude-spec-consumer`

其中 `005-claude-spec-consumer` 已经完成第一轮 backend-driven 闭环。

---

## 当前还不支持什么

项目虽然已经能跑，但还没有完成这些能力：

### 1. 还不支持真正的全自动链式工作流
当前还是按 stage 自动消费，不能做到“用户给需求后一路自动跑完整条流程”。

### 2. 还没有打通 implement 阶段的真实 backend 自动化
现在打通的是早期文本型阶段，代码实现阶段还没真正接入。

### 3. consumer 还不是完全 bundle-native
现在虽然已经有 stage bundle，但 consumer 仍有部分 stage-specific 逻辑。

### 4. 还没有自动 retrospective 流程
retrospective 已经开始有文档，但还没有正式接进 workflow。

---

## 当前项目所处阶段

可以把这个项目当前阶段理解为：

> **轻量可运行的 SDD / agentic workflow 原型系统**

或者更具体一点：

> **已经具备 runtime wiring、prompt layer、execution bundles，以及 early-stage backend consumption 的轻量工作流系统。**

它已经不是一个“概念项目”，但也还不是一个“全自动工程平台”。

---

## 下一步最值得做的方向

当前最自然的下一步包括：

1. 做 early-stage auto flow（前半段自动串行）
2. 让 backend consumer 更直接依赖 bundle
3. 强化 validation
4. 接入 retrospective 流程
5. 再跑不同类型的真实 feature
6. 最后再考虑 implement 阶段自动化

---

## 适合谁看这个项目

这个项目适合这些人：

- 想做 AI-native 软件工程工作流的人
- 想把 SDD 落到真实仓库的人
- 正在探索 agentic coding pipeline 的团队
- 没有复杂 Agent Teams，但已经有 Skills / Subagents / Hooks 的环境
- 希望把 AI 编程从“prompt 工具”推进成“阶段化工程系统”的人

---

## 一句话总结

> `agentic-sdd-pipeline` 是一个把 SDD、runtime、prompt layer 和真实 backend consumption 结合起来的轻量 agentic 编程系统原型，用来探索 AI 如何以更工程化的方式完成早期软件开发流程。
