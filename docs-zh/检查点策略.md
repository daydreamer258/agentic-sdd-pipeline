# 检查点策略

## 1. 概述

本文档定义检查点策略 - 在实现阶段创建审批暂停点的机制。

---

## 2. 什么是检查点

检查点是工作流中的暂停点，需要人工审批才能继续。

它确保：
- 重大变更经过审查
- 风险被正确评估
- 人类保持控制

---

## 3. 检查点触发条件

### 3.1 自动触发

以下情况自动创建检查点：
- 风险评估为"中"或"高"
- 修改核心架构
- 更改API接口

### 3.2 手动触发

开发者可以手动请求检查点：
- 当不确定时
- 当需要额外审查时

---

## 4. 检查点流程

```
实现阶段开始
        |
        v
   风险评估
        |
        +-- 低风险 --> 继续执行
        |
        +-- 中/高风险 --> 创建检查点
                                |
                                v
                         通知审批者
                                |
                    +-- 审批 --> 继续执行
                    |
                    +-- 拒绝 --> 停止/修改
```

---

## 5. 检查点信息

### 5.1 包含内容

检查点报告包含：
- 功能描述
- 风险评估
- 实现计划
- 更改的文件
- 依赖影响

### 5.2 格式

```markdown
# 检查点: <feature-name>

## 功能
<描述>

## 风险评估
- 级别: <low/medium/high>
- 原因: <原因>

## 实现计划
<计划>

## 更改的文件
- <file1>
- <file2>

## 审批
[等待审批 / 已批准 / 已拒绝]
```

---

## 6. 审批标准

### 6.1 批准标准

- 风险评估合理
- 实现计划合理
- 影响已评估
- 没有明显的安全问题

### 6.2 拒绝原因

- 风险过高
- 计划不充分
- 影响范围不清楚

---

## 7. 实施

### 7.1 实现脚本

`implement-checkpoint.sh` 管理检查点：

```bash
# 创建检查点
./scripts/implement-checkpoint.sh create <feature-dir>

# 检查是否有待处理检查点
./scripts/implement-checkpoint.sh status <feature-dir>

# 批准检查点
./scripts/implement-checkpoint.sh approve <feature-dir>

# 拒绝检查点
./scripts/implement-checkpoint.sh reject <feature-dir>
```

### 7.2 状态存储

检查点状态存储在 `state.json` 中：

```json
{
  "needs_review": true,
  "checkpoint_status": "pending"
}
```

---

## 8. 最佳实践

1. **及时响应** - 尽快处理检查点
2. **提供上下文** - 包含足够的信息
3. **记录决定** - 记录批准/拒绝的原因
