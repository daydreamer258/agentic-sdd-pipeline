# 提示词层

## 1. 概述

本文档描述了流水线的提示词层架构 - 定义如何为AI后端构建提示词。

---

## 2. 提示词层架构

```
┌──────────────────────────────────────────────┐
│                 提示词源                       │
├──────────────────┬───────────────────────────┤
│    Skills        │     Subagents             │
│  (流程规则)      │     (角色定义)             │
└────────┬─────────┴────────────┬──────────────┘
         │                      │
         └──────────┬───────────┘
                    │
                    v
         ┌──────────────────────┐
         │    模板文件           │
         │  (制品结构)           │
         └──────────┬───────────┘
                    │
                    v
         ┌──────────────────────┐
         │    输入制品           │
         │  (上下文)             │
         └──────────┬───────────┘
                    │
                    v
         ┌──────────────────────┐
         │    完整提示词          │
         └──────────────────────┘
```

## 3. 提示词组件

### 3.1 Skills

定义**如何**执行每个阶段：

- 阶段意图
- 行为规则
- 完成标准
- 升级条件

示例：`skills/sdd-spec/SKILL.md`

### 3.2 Subagents

定义**谁**执行工作：

- 角色描述
- 职责范围
- 专业领域

示例：`subagents/spec-writer.md`

### 3.3 模板

定义**产出什么**：

- 制品结构
- 必要部分
- 格式指南

示例：`templates/01-spec.md`

### 3.4 输入制品

提供**上下文**：

- 上一阶段的产出
- 项目级规则
- 任何相关文档

## 4. 提示词构建过程

### 4.1 构建步骤

```
1. 选择正确的Skill
2. 选择正确的Subagent
3. 选择正确的模板
4. 读取相关输入制品
5. 组合所有部分
6. 添加系统指令
7. 生成完整提示词
```

### 4.2 提示词组装

```sh
# 伪代码
PROMPT=""
PROMPT+=$(cat $SKILL_PATH)
PROMPT+="\n\n"
PROMPT+=$(cat $SUBAGENT_PATH)
PROMPT+="\n\n"
PROMPT+=$(cat $TEMPLATE_PATH)
PROMPT+="\n\n"
PROMPT+="## Context\n"
PROMPT+=$(cat $INPUT_ARTIFACTS)
```

## 5. 技能提示词示例

### sdd-spec 技能提示词

```markdown
# sdd-spec

## Purpose
Turn 00-intake.md into a behavior-oriented 01-spec.md.

## When to use
Use this Skill when the pipeline is at the spec stage.

## Inputs
Read:
- features/<feature>/00-intake.md
- templates/01-spec.md

## Outputs
Write:
- features/<feature>/01-spec.md

## Required output
The spec must include:
- problem statement
- users / actors
- desired behavior
- acceptance criteria
- edge cases
- non-goals
- assumptions

## Behavioral rules
- Focus on what the system should do and why
- Keep the spec behavior-oriented
- Make success criteria explicit
- Include non-goals to prevent scope drift
- Only use implementation details when necessary for precision

## Must avoid
- Writing a technical plan inside the spec
- Adding architecture decisions too early
- Broadening feature scope beyond intake without explicit note

## Escalate when
Escalate instead of guessing if:
- User behavior remains materially ambiguous
- Success criteria cannot be made concrete
- Feature scope appears unstable or contradictory
```

## 6. 子代理提示词示例

### spec-writer 子代理

```markdown
# Spec Writer

You are an expert at writing clear, behavior-oriented specifications.

## Your role
Transform raw requirements into precise, testable specifications.

## Expertise
- Requirements analysis
- Behavior specification
- Test criteria definition
- Edge case identification

## Responsibilities
- Write clear problem statements
- Define measurable success criteria
- Identify all relevant edge cases
- Specify non-goals explicitly

## Working style
- Be precise and unambiguous
- Focus on "what" not "how"
- Include rationale for decisions
- Test your spec by seeing if a developer can implement from it
```

## 7. 提示词优化

### 7.1 优化原则

1. **清晰** - 避免歧义
2. **具体** - 提供具体例子
3. **可操作** - 告诉AI具体怎么做
4. **有边界** - 明确什么不该做

### 7.2 常见问题

| 问题 | 解决 |
|------|------|
| 输出太宽泛 | 添加具体例子 |
| 忽略约束 | 明确列出约束 |
| 过度工程 | 明确"不要做什么" |
| 遗漏边界情况 | 提示检查边界 |

## 8. 提示词版本管理

### 8.1 版本策略

- 初始版本：v1
- 改进版本：v2, v3, ...
- 不兼容更改：v2.0

### 8.2 更新流程

1. 测试新提示词
2. 验证输出质量
3. 更新版本号
4. 记录更改

---

## 9. 测试提示词

### 9.1 测试方法

1. 运行实际阶段
2. 检查输出质量
3. 验证与模板的一致性

### 9.2 质量指标

- 完整性：是否包含所有必需部分？
- 正确性：信息是否准确？
- 一致性：是否与上一阶段一致？

---

## 10. 最佳实践

1. **保持简洁** - 提示词不应太长
2. **使用示例** - 具体例子胜过描述
3. **明确边界** - 告诉AI什么不该做
4. **版本化** - 跟踪提示词更改
5. **测试** - 验证提示词效果
