# 技能接口

## 1. 概述

本文档定义技能（Skills）的接口规范。

技能是编码可复用流程行为的提示词集合。它们为代理提供：
- 阶段特定的指导
- 质量标准
- 升级规则
- 边界定义

---

## 2. 技能结构

每个技能应包含以下部分：

```markdown
# <skill-name>

## Purpose
（目的 - 一句话说明这个技能是做什么的）

---

## When to use
（何时使用 - 触发条件和场景）

---

## Inputs
（输入 - 需要读取什么制品）

---

## Outputs
（输出 - 需要产生什么制品）

---

## Required output
（必需输出 - 输出必须包含什么）

---

## Behavioral rules
（行为规则 - 如何执行工作）

---

## Must avoid
（必须避免 - 什么是不应该做的）

---

## Escalate when
（升级条件 - 何时应该升级而不是继续）

---

## Definition of done
（完成定义 - 怎么才算完成）
```

## 3. sdd-intake 技能

```yaml
Purpose: 将原始请求转化为结构化需求收集

When to use:
  - 用户提出新功能请求时
  - 需要澄清模糊需求时

Inputs:
  - 原始用户请求
  - 项目级规则（如果有）

Outputs:
  - 00-intake.md

Required output:
  - 请求摘要
  - 已知的约束
  - 未知/问题
  - 初始成功定义
  - 初始假设

Behavioral rules:
  - 准确总结用户意图
  - 识别所有明显的约束
  - 明确记录所有假设
  - 如果需要澄清，说明需要什么

Must avoid:
  - 假设用户未说明的内容
  - 跳过不确定的部分

Escalate when:
  - 核心目标完全模糊
  - 存在冲突的约束

Definition of done:
  - 请求被清晰总结
  - 所有约束和假设已记录
```

## 4. sdd-spec 技能

```yaml
Purpose: 将需求收集转化为行为导向的规格

When to use:
  - intake阶段完成时
  - 需要编写规格时

Inputs:
  - 00-intake.md
  - templates/01-spec.md

Outputs:
  - 01-spec.md

Required output:
  - 问题陈述
  - 用户/角色
  - 预期行为
  - 验收标准
  - 边界情况
  - 非目标
  - 假设

Behavioral rules:
  - 关注系统应该做什么以及为什么
  - 保持规格面向行为
  - 使成功标准明确
  - 包含非目标以防止范围漂移
  - 仅在精确性需要时使用实现细节

Must avoid:
  - 在规格中写技术计划
  - 过早加入架构决策
  - 在没有明确说明的情况下扩大功能范围

Escalate when:
  - 用户行为仍然实质性模糊
  - 成功标准无法具体化
  - 功能范围不稳定或矛盾

Definition of done:
  - 规格足够清晰，计划者可以在不推断核心行为的情况下设计实现
```

## 5. sdd-plan 技能

```yaml
Purpose: 将规格转化为技术实现计划

When to use:
  - spec阶段完成并批准时
  - 需要创建技术计划时

Inputs:
  - 01-spec.md
  - 项目架构原则

Outputs:
  - 02-plan.md
  - 可选的 03-research.md, 04-data-model.md, 05-contracts.md

Required output:
  - 架构概要
  - 主要组件/模块
  - 技术栈选择
  - 约束和理由
  - 验证方法
  - 运营/安全/性能考虑
  - 已知风险

Behavioral rules:
  - 定义功能应如何构建
  - 使约束明确
  - 包含验证策略
  - 记录所有重大决策

Must avoid:
  - 更改规格中定义的行为
  - 忽略非目标

Escalate when:
  - 架构依赖于未解决的产品决策
  - 所需集成未知

Definition of done:
  - 计划足够详细，任务分解者可以创建可执行的任务
```

## 6. sdd-tasks 技能

```yaml
Purpose: 将计划转化为可执行的任务列表

When to use:
  - plan阶段完成时
  - 需要分解任务时

Inputs:
  - 01-spec.md
  - 02-plan.md
  - templates/06-tasks.md

Outputs:
  - 06-tasks.md

Required output:
  - 有序任务列表
  - 每个任务的描述
  - 依赖关系
  - 并行标记（适用时）
  - 每个任务的验证提示

Behavioral rules:
  - 创建可审查的任务
  - 使任务足够小以独立实现
  - 标记依赖关系
  - 识别可并行的工作

Must avoid:
  - 创建等同于史诗的大型任务
  - 创建无法独立验证的任务

Escalate when:
  - 任务仍然是史诗规模
  - 实现边界不清晰

Definition of done:
  - 任务可以按顺序执行
  - 每个任务都有清晰的边界
  - 依赖关系明确
```

## 7. sdd-implement 技能

```yaml
Purpose: 在有边界的范围内实现任务

When to use:
  - tasks阶段完成时
  - 需要实现特定任务时

Inputs:
  - 06-tasks.md (相关任务)
  - 01-spec.md
  - 02-plan.md
  - 仓库上下文

Outputs:
  - 代码更改
  - 07-implementation-log.md

Required output:
  - 已实现的任务
  - 更改的文件列表
  - 重要决策记录
  - 本地验证结果

Behavioral rules:
  - 在分配的范围内工作
  - 遵循项目编码标准
  - 运行适当的本地检查
  - 记录重要决策

Must avoid:
  - 扩展任务范围
  - 修改上游制品
  - 引入未计划的架构变更

Escalate when:
  - 实现需要更改上游假设
  - 任务分解不可执行

Definition of done:
  - 任务按规格实现
  - 代码通过本地验证
  - 实现日志完整
```

## 8. sdd-validate 技能

```yaml
Purpose: 验证实现是否符合质量和一致性期望

When to use:
  - 实现阶段完成时
  - 需要验证变更时

Inputs:
  - 更改的文件
  - 01-spec.md
  - 02-plan.md
  - 06-tasks.md
  - 07-implementation-log.md

Outputs:
  - 08-validation.md

Required output:
  - 运行的检查
  - 测试/构建/代码检查结果
  - 规格一致性发现
  - 计划一致性发现
  - 漂移或质量问题
  - 通过/失败建议

Behavioral rules:
  - 验证代码质量（测试、代码检查）
  - 验证规格一致性
  - 验证计划一致性
  - 识别任何偏差

Must avoid:
  - 仅检查测试通过
  - 忽略规格或计划

Escalate when:
  - 验证揭示规格/计划级问题
  - 存在系统性质量问题

Definition of done:
  - 所有三个轴都被验证
  - 有明确的通过/失败建议
```

## 9. 技能使用指南

### 调用技能

当代理进入特定阶段时：

1. 加载该阶段的技能
2. 阅读输入制品
3. 根据技能指导执行
4. 产生输出制品

### 遵循技能

- 技能是规范，不是建议
- 违反技能应有充分理由
- 遇到升级条件时升级

### 更新技能

当发现技能不足时：
1. 记录具体问题
2. 提出改进建议
3. 更新技能文档

---

## 10. 技能版本管理

技能可能有版本。版本格式：`skill-name/vN`

- v1：初始版本
- v2：改进版本
- 不兼容的更改应增加主版本号

## 11. 技能质量检查

在创建或更新技能时，检查：

- [ ] 有清晰的Purpose
- [ ] 明确When to use条件
- [ ] 定义Inputs和Outputs
- [ ] 列出Required output
- [ ] 包含Behavioral rules
- [ ] 列出Must avoid
- [ ] 定义Escalate when条件
- [ ] 有清晰的Definition of done
