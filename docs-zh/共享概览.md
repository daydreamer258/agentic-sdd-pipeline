# 共享概览

## 什么是规格驱动开发 (SDD)？

规格驱动开发（Spec-Driven Development，SDD）是一种软件开发方法论，它将**规格（Spec）**置于开发过程的中心。

### 核心原则

1. **规格优先** - 在编写代码之前先编写规格
2. **行为导向** - 规格描述系统应该做什么，而不是怎么做
3. **可验证** - 规格应有明确的验收标准
4. **迭代 refinement** - 规格随着理解深入而演进

---

## 为什么需要SDD？

### AI编码的挑战

现代AI编码代理可以：
- 探索代码库
- 生成代码
- 迭代调试
- 跨文件应用补丁
- 运行测试并对失败做出反应

但它们仍然困难的是：
- 需求模糊
- 架构隐式
- 约束分散
- 任务太大
- 缺少验证门控

### SDD如何提供帮助

SDD将工作流从：

```
提示词 -> 代码 -> 之后修补问题
```

转变为：

```
意图 -> 规格 -> 计划 -> 任务 -> 实现 -> 验证
```

## SDD的关键概念

### 1. 制品 (Artifacts)

每个阶段的产出：

| 阶段 | 制品 | 目的 |
|------|------|------|
| 需求收集 | 00-intake.md | 捕获原始请求 |
| 规格 | 01-spec.md | 定义行为和验收标准 |
| 计划 | 02-plan.md | 技术架构和约束 |
| 任务 | 06-tasks.md | 可执行的工作单元 |
| 实现 | 07-implementation-log.md | 做了什么 |
| 验证 | 08-validation.md | 是否满足要求 |

### 2. 阶段门控 (Stage Gates)

每个阶段必须完成才能进入下一阶段：
- spec 需要 intake 完成
- plan 需要 spec 批准
- tasks 需要 spec + plan
- implement 需要 tasks
- validate 需要实现日志

### 3. 人在回路 (Human-in-the-loop)

关键点需要人工审批：
- 规格编写后
- 计划批准后
- 实现开始前
- 验证后，合并前

## 与传统开发的对比

| 方面 | 传统开发 | SDD |
|------|----------|-----|
| 需求 | 口头/文档 | 结构化制品 |
| 计划 | 口头 | 书面计划 |
| 实现 | 边做边改 | 按计划执行 |
| 验证 | 测试 | 多轴验证 |
| 追溯 | 困难 | 完整链路 |

---

## SDD的优势

### 1. 更清晰的意图
规格强制明确需求，减少误解。

### 2. 更好的可追溯性
从实现到规格的完整追溯。

### 3. 更早发现问题
阶段门控在问题变贵之前捕获。

### 4. 更好的协作
制品是共享的理解基础。

### 5. 更容易维护
规格是系统的真实来源。

---

## 何时使用SDD

SDD特别适合：

- **复杂功能** - 需要多个迭代
- **多人协作** - 需要共享理解
- **高风险变更** - 需要严格审查
- **AI辅助开发** - 需要结构化引导

可能过度的情况：

- **简单修复** - 快速补丁
- **原型探索** - 快速验证想法
- **紧急修复** - 时间敏感

## 实施建议

### 从小开始
- 从一个feature开始
- 逐渐扩展到更多

### 保持灵活
- 不是每个阶段都是强制的
- 根据需要调整

### 关注价值
- 规格应该帮助，而不是阻碍
- 避免过度工程

### 持续改进
- 从每个feature学习
- 调整流程和模板

## 常见陷阱

### 1. 规格过度
详细规格但从不使用。

**解决**：保持规格足够详细以做出决定，但不是一切都详细。

### 2. 规格与实现脱节
实现不遵循规格。

**解决**：验证阶段应明确检查一致性。

### 3. 计划过度
详细计划但无法执行。

**解决**：任务应足够小且可执行。

### 4. 跳过阶段
为了速度跳过规格或计划。

**解决**：阶段门控确保纪律。
