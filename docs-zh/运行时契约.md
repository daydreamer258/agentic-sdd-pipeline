# 运行时契约

## 1. 概述

本文档定义了流水线各组件之间的运行时契约。这些契约确保各部分可以正确地协同工作。

---

## 2. 阶段契约

### 2.1 输入契约

每个阶段需要特定输入：

| 阶段 | 必需输入 |
|------|----------|
| intake | (无，需要原始请求) |
| spec | `00-intake.md` |
| plan | `01-spec.md` |
| tasks | `01-spec.md`, `02-plan.md` |
| implement | `01-spec.md`, `02-plan.md`, `06-tasks.md` |
| validate | `01-spec.md`, `02-plan.md`, `06-tasks.md`, `07-implementation-log.md` |

### 2.2 输出契约

每个阶段产生特定输出：

| 阶段 | 必需输出 |
|------|----------|
| intake | `00-intake.md` |
| spec | `01-spec.md` |
| plan | `02-plan.md` |
| tasks | `06-tasks.md` |
| implement | 代码更改, `07-implementation-log.md` |
| validate | `08-validation.md` |

## 3. 状态契约

### 3.1 state.json 字段

```json
{
  "feature_id": "string",
  "feature_slug": "string",
  "feature_title": "string",
  "current_stage": "string",
  "status": "string",
  "last_artifact": "string",
  "next_stage": "string",
  "last_updated_at": "ISO8601 timestamp",
  "validation_result": "string",
  "needs_review": "boolean"
}
```

### 3.2 有效状态值

- `current_stage`: `intake`, `spec`, `plan`, `tasks`, `implement`, `validate`, `done`
- `status`: `pending`, `in_progress`, `completed`, `blocked`, `needs_review`

### 3.3 状态转换规则

```
pending -> in_progress (进入阶段)
in_progress -> completed (制品完成)
in_progress -> blocked (遇到阻塞)
blocked -> in_progress (解除阻塞)
completed -> pending (进入下一阶段)
```

## 4. 制品契约

### 4.1 命名规范

| 制品 | 命名格式 |
|------|----------|
| 需求收集 | `00-intake.md` |
| 规格 | `01-spec.md` |
| 计划 | `02-plan.md` |
| 研究 | `03-research.md` |
| 数据模型 | `04-data-model.md` |
| 契约 | `05-contracts.md` |
| 任务 | `06-tasks.md` |
| 实现日志 | `07-implementation-log.md` |
| 验证 | `08-validation.md` |
| 回顾 | `09-retrospective.md` |

### 4.2 制品内容要求

每个制品应包含：
- 清晰的目标陈述
- 充分的上下文
- 明确的完成标准

### 4.3 制品质量门槛

| 制品 | 最低质量要求 |
|------|--------------|
| intake | 请求被总结，约束已知 |
| spec | 成功标准明确，边界情况列出 |
| plan | 架构清晰，约束明确 |
| tasks | 任务可执行，依赖明确 |
| implement | 代码更改已记录 |
| validate | 检查已运行，建议已给出 |

## 5. Hook契约

### 5.1 Hook接口

每个hook脚本应接受：
- `$1`: feature目录路径
- `$2`: 阶段名称（除 `on_feature_init`）

### 5.2 Hook返回值

- `0`: 成功，允许继续
- `非0`: 失败，阻止操作

### 5.3 标准Hooks

| Hook | 参数 | 预期行为 |
|------|------|----------|
| `on_feature_init` | feature-dir | 初始化scaffolding |
| `before_stage_transition` | feature-dir, stage | 验证前置条件 |
| `before_implement` | feature-dir | 风险评估 |
| `after_artifact_write` | feature-dir, artifact | 后期处理 |
| `after_validation` | feature-dir | 验证后处理 |

---

## 6. 处理器契约

### 6.1 处理器接口

每个处理器应：
1. 验证前置条件
2. 创建必要的目录
3. 调用 `build-stage-bundle.sh`
4. 输出就绪状态

### 6.2 处理器输出格式

```
STAGE_READY=<stage>
BUNDLE_PATH=<path-to-bundle>
```

---

## 7. Bundle契约

### 7.1 Bundle结构

```markdown
# Stage Bundle: <stage>

## Current State
- Feature: <name>
- Current Stage: <stage>
- Next Stage: <next>

## Goal
<描述此阶段的目标>

## Inputs
- <artifact1>
- <artifact2>

## Outputs
- <output-artifact>

## Skill
- Path: <skill-path>
- Usage: <如何使用>

## Subagent
- Path: <subagent-path>
- Role: <角色描述>

## Execution Suggestion
<如何执行此阶段的建议>
```

### 7.2 Bundle生成

```sh
./scripts/build-stage-bundle.sh <feature-dir> <stage>
```

输出：`features/<id>-<slug>/.runtime/<stage>.bundle.md`

---

## 8. 后端消费者契约

### 8.1 消费者接口

每个消费者应：
1. 读取输入制品
2. 加载Skill和Subagent提示词
3. 生成完整提示词
4. 调用后端
5. 写出输出制品
6. 返回退出码

### 8.2 消费者输出

- 成功：写出制品，返回0
- 失败：报错，返回非0

### 8.3 支持的后端

| 后端 | 阶段支持 |
|------|----------|
| Claude Code | intake, spec, plan, tasks, implement, validate |

## 9. 命令行接口契约

### 9.1 标准命令

```bash
# 初始化
./scripts/init-feature.sh <id> <slug> [title]

# 执行阶段
./scripts/run-stage.sh <feature-dir> <stage>
./scripts/execute-stage.sh <feature-dir> <stage>

# 自动工作流
./scripts/auto-workflow.sh <feature-dir> full [start] [end]
./scripts/auto-workflow.sh <feature-dir> single <stage>
./scripts/auto-workflow.sh <feature-dir> range <start> <end>

# 消费者
./scripts/consume-stage-with-claude.sh <feature-dir> <stage>

# 完成
./scripts/complete-artifact.sh <feature-dir> <artifact>

# 状态
./scripts/feature-summary.sh <feature-dir>
```

### 9.2 退出码

- `0`: 成功
- `非0`: 失败

---

## 10. 违规处理

### 10.1 违反输入契约

如果必需输入不存在：
- 报告错误
- 阻止进入阶段
- 建议需要的制品

### 10.2 违反输出契约

如果输出不满足质量门槛：
- 报告质量问题
- 不标记为完成
- 建议改进

### 10.3 违反状态契约

如果状态不一致：
- 尝试自动修复
- 如果无法修复，报告错误
