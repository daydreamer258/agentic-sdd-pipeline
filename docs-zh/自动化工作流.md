# 自动化工作流

## 1. 目的

本文档描述了SDD流水线的轻量级自动化工作流运行器。

目标是同时支持：

- 端到端早期阶段自动化
- 单个阶段或阶段范围的定向执行

---

## 2. 支持的模式

工作流运行器支持三种模式：

### `full`
自动运行连续的工作流。

```sh
./scripts/auto-workflow.sh <feature-dir> full [start-stage] [end-stage]
```

默认值：
- 开始阶段：`intake`
- 结束阶段：`tasks`

### `single`
运行一个特定阶段。

```sh
./scripts/auto-workflow.sh <feature-dir> single <stage>
```

### `range`
运行一个阶段区间。

```sh
./scripts/auto-workflow.sh <feature-dir> range <start-stage> <end-stage>
```

---

## 3. 当前阶段支持

当前自动化覆盖：

- `intake` ✅
- `spec` ✅
- `plan` ✅
- `tasks` ✅
- `implement` ✅（有边界/实验性）
- `validate` ✅

---

## 4. 执行模型

对于每个阶段，运行器执行：

1. `execute-stage.sh`
2. 后端消费（如果已配置）
3. `complete-artifact.sh`
4. 当阶段是`validate`时调用`after_validation.sh`

这创建了仓库中第一个真正的链接执行路径。

---

## 5. 示例

### 完整早期阶段流程

```sh
./scripts/auto-workflow.sh ./features/006-auto-workflow full intake tasks
```

### 仅运行一个阶段

```sh
./scripts/auto-workflow.sh ./features/005-claude-spec-consumer single validate
```

### 运行一个范围

```sh
./scripts/auto-workflow.sh ./features/005-claude-spec-consumer range plan tasks
```

---

## 6. 当前限制

当前自动化工作流有意保持轻量。

它尚未：

- 在当前轻量模式之外可靠地验证实现质量
- 在后端故障时恢复（除退出并报错外）
- 通过共享适配器层支持多个后端
- 为implement应该自动运行还是需要审批提供完善的检查点策略

---

## 7. 为什么重要

在这个运行器存在之前，仓库支持阶段级自动化，但没有链接的工作流执行。

有了`auto-workflow.sh`，项目现在支持：

- 整个早期阶段运行
- 定向阶段执行
- 阶段范围执行

这是迈向更真正的智能流水线的第一步。
