# 系统设计

## 1. 概述

本文档为围绕**规格驱动开发(SDD)**构建的**智能编程流水线**提出了一种实用的**v1系统设计**。

目标环境是受限的：

- Skills（技能）可用
- Subagents（子代理）可用
- Agent Teams（代理团队）不可用

因此，设计采用**单一编排器 + 阶段专用技能 + 专业子代理**架构。

---

## 2. 设计目标

系统应该：

1. 将模糊请求转化为分阶段的工程制品
2. 使AI编码工作比原始的提示词到代码工作流更可靠
3. 支持从实现到任务、计划、规格的可追溯性
4. 让人类控制主要的状态转换
5. 在编排原语仅限于Skills + Subagents的环境中工作
5. 通过验证反馈和回顾持续改进

---

## 3. v1非目标

系统**不**试图：

- 实现完整的规格即源码开发
- 消除人工审查
- 自主合并到生产环境
- 模拟复杂的代理团队拓扑
- 用一个相同的工作流解决所有项目规模
- 为所有组织构建一个完美的通用标准

v1有意聚焦于**规范的、可审查的、生产级的工作流**。

---

## 4. 核心架构

```
用户 / 产品请求
        |
        v
主编排器
  ├─ 需求收集 / 澄清 Skill
  ├─ 规格编写 Subagent
  ├─ 计划 / 架构 Subagent
  ├─ 任务分解 Subagent
  ├─ 实现 Subagent(s)
  ├─ 审查 / 验证 Subagent
  └─ 回顾 / 反馈阶段
        |
        v
制品存储（feature文件夹 + 项目级流程记忆）
```

### 4.1 主编排器职责

编排器是系统的控制平面。

职责：

- 接收原始请求
- 确定当前阶段
- 应用该阶段正确的Skill
- 在需要时生成正确的子代理
- 在进入阶段前验证所需输入是否存在
- 决定是继续、阻塞还是升级给人类
- 保持制品链的一致性
- 记录决策和状态转换

编排器**不应该**是大任务的主要重负荷实现者。它的价值在于工作流控制、制品规范和审查路由。

---

## 5. 执行模型

v1系统使用**阶段门控执行模型**。

### 流水线阶段

1. 需求收集 / 澄清
2. 规格编写
3. 计划
4. 任务分解
5. 实现
6. 验证 / 审查
7. 回顾 / 反馈

每个阶段都有：

- 必需输入
- 预期输出
- 完成定义
- 转换规则
- 升级条件

### 为什么需要阶段门控？

这避免了代理直接从请求跳到实现的常见失败模式：

- 澄清假设
- 定义架构
- 约束范围
- 准备验证标准

---

## 6. 状态模型

功能/变更请求应建模为有状态的单元。

建议的状态：

- `intake_pending` - 待收集需求
- `clarification_needed` - 需要澄清
- `spec_drafting` - 编写规格中
- `spec_review` - 规格审查中
- `plan_drafting` - 编写计划中
- `plan_review` - 计划审查中
- `tasks_drafting` - 编写任务中
- `tasks_review` - 任务审查中
- `implementation_ready` - 准备实现
- `implementing` - 实现中
- `validation_pending` - 待验证
- `validation_failed` - 验证失败
- `revision_required` - 需要修订
- `validated` - 已验证
- `retrospective_pending` - 待回顾
- `closed` - 已关闭

### 状态转换原则

如果必需的制品缺失或质量不达标，阶段不应推进。

示例：

- 没有`01-spec.md` → 不能进入计划阶段
- 没有已批准的`02-plan.md` → 不能进入任务分解
- 没有`06-tasks.md` → 不能开始多任务实现
- 验证失败 → 必须返回实现或修复上游制品

---

## 7. 人在回路检查点

v1应在主要转换点明确保留人工检查点。

建议的审批点：

1. 规格编写完成后
2. 计划编写完成后
3. 实现开始前
4. 验证后，合并/关闭前

### 为什么重要

人工审查是捕获以下问题最高效的地方：

- 错误的问题定义
- 缺失的非目标
- 危险的架构假设
- 范围过大的任务计划
- 通过测试但违反意图的实现

---

## 8. 制品为中心的控制平面

制品是系统的支柱。

编排器应将制品视为系统状态，而不仅仅是聊天历史。

v1最小制品集：

- `00-intake.md`
- `01-spec.md`
- `02-plan.md`
- `06-tasks.md`
- `07-implementation-log.md`
- `08-validation.md`
- `09-retrospective.md`

可选但非常有用的：

- `03-research.md`
- `04-data-model.md`
- `05-contracts.md` 或 `contracts/`

编排器应始终优先读取当前制品，而不是依赖冗长的对话记忆。

---

## 9. 子代理角色

### 9.1 规格编写者

输入：
- 需求收集制品
- 澄清内容
- 项目级规则

输出：
- 规格草案

重点：
- 行为
- 成功标准
- 边界情况
- 非目标

### 9.2 计划 / 架构师

输入：
- 已批准或接近批准的规格
- 全局架构规则
- 技术约束

输出：
- 实现计划
- 可选的模型/契约/研究

重点：
- 架构
- 边界
- 技术栈选择
- 运营约束
- 验证策略

### 9.3 任务分解者

输入：
- 规格 + 计划

输出：
- 有序任务列表

重点：
- 任务粒度
- 依赖关系
- 可并行工作
- 可审查的增量

### 9.4 实现者

输入：
- 任务子集
- 相关制品
- 仓库上下文

输出：
- 代码更改
- 实现笔记

重点：
- 有边界的执行
- 任务一致性
- 本地验证

### 9.5 审查 / 验证者

输入：
- 更改的文件
- 规格
- 计划
- 任务

输出：
- 验证报告

重点：
- 规格一致性
- 计划一致性
- 漂移检测
- 质量和安全检查

---

## 10. 技能模型

Skills应编码可复用的流程层。

每个阶段专用的Skill应定义：

- 触发条件
- 必需输入
- 输出模板
- 最低完整性检查
- 需要避免的内容
- 何时升级

建议的v1 Skills：

- `sdd-intake`
- `sdd-spec-writer`
- `sdd-plan-architect`
- `sdd-task-decomposer`
- `sdd-implementer`
- `sdd-review-validator`
- `sdd-retrospective`

编排器使用Skills来约束行为，使用Subagents来执行工作。

---

## 11. 验证模型

v1中的验证不应仅限于测试。

审查/验证者应评估至少3个轴：

1. **实现正确性**
   - 测试
   - 代码检查
   - 构建成功

2. **规格一致性**
   - 实现是否满足预期行为？
   - 是否覆盖了成功标准？
   - 是否遵守了非目标？

3. **计划一致性**
   - 实现是否尊重架构和约束？
   - 是否引入了漂移或未经批准的模式？

只验证测试的系统仍会积累架构和行为漂移。

---

## 12. 故障处理

系统应区分故障类别。

### A类：实现故障
示例：
- 测试失败
- 代码损坏
- 任务未完成

操作：
- 返回实现阶段

### B类：计划故障
示例：
- 选用的方法与约束冲突
- 架构假设无效

操作：
- 重新开放计划

### C类：规格故障
示例：
- 缺失边界情况
- 意图模糊
- 下游持续误解

操作：
- 重新开放规格阶段

### D类：流程故障
示例：
- 任务粒度持续错误
- 子代理反复越界
- 验证每次运行都捕获相同类型的缺陷

操作：
- 更新Skills、模板和流程记忆

---

## 13. 为什么这个设计适合v1

这个设计有意保守。

它倾向于：

- 明确的状态
- 明确的制品
- 人工检查点
- 简单的编排
- 可复用的流程约束

而不是：

- 高度自主的图执行
- 推测性的团队级委托
- 隐藏的状态转换
- 自由形式的多代理即兴表演

这种权衡是合适的，因为第一个难题不是原始的自主性。
第一个难题是**使工作流可检查、可调试和可信任**。

---

## 14. 开放的设计问题

这些问题应在下次计划中解决：

1. 阶段之间的确切审批模型是什么？
2. 制品模板应该多严格？
3. 每个feature文件夹中什么是强制的 vs 可选的？
4. 项目级章程/规则将如何存储？
5. 实现任务应如何分组以并行执行？
6. 验证结果应如何反馈到全局流程记忆？
7. 什么应持久化在仓库中 vs 外部运行时内存中？
